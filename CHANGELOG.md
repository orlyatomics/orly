- **Commutative global-merge fast-lane** (#234, behind `--tetris_commutative_fastlane`, **default off**). The single-threaded global-POV Tetris merge re-snapshotted *every* waiting child each round but promoted only one (a 2014 placeholder), so draining N queued child POVs was O(NÂ²) re-snapshot. The fast-lane promotes **all** assertion-free (commutative `+=`/`|=`) children per round, each in its own transaction, so the one-Pusher-per-repo invariant and each child's session-id metadata (load-bearing for replication notifications) are preserved exactly; assertion-bearing RMW children keep the one-per-round discipline byte-for-byte. New reporter counter `Tetris Children Considered / Push` makes the quadratic observable: it drops from ~30 to 1.0 at 32 child POVs. Shipped default-off (PR #236). **The default-on flip was investigated and declined** (this PR): the Â§8.6 prerequisites were resolved â€” the pre-existing race the fast-lane amplifies is fixed (#237, below) and write backpressure was added (below) â€” but a K=24 release soak (both synchronous and pipelined, on the Tetris path) measured the flag throughput- *and* sustainability-neutral: with it on or off the run degrades identically, because the binding concurrent-write constraint is the per-write acceptance path / fast-scheduler capacity, not the merge tournament (the merge keeps up; backlog â‰ˆ1.3 children/round). So the flag stays off, available for the genuine deep-backlog regime. The benchmark (`examples/agent-swarm/concurrent_bench.py`, `run-concurrent-bench.sh`) ships with it. See `docs/design/concurrent-merge-throughput.md` Â§8.
- **Write backpressure on the merge backlog** (#234, `--tetris_backpressure_threshold`, default 50000; 0 disables). When a writer's POV child repo has more than the threshold of un-promoted updates backed up in its memtable, the accept path cooperatively yields its fiber (`Fiber::YieldSlow`) after committing each write until the global merge drains the backlog below the watermark. So when accept outruns promote, sustained writers degrade to a graceful slowdown that paces accept to promote, instead of the memtable growing until `bad_alloc`. A safety net for the genuine deep-backlog regime; harmless and inert under normal load (the threshold is far above the steady-state backlog).
- **`union_map` sequence operator** (#219 phase 2). `seq union_map elem` maps each element of a sequence to a set and unions the results, from an implicit empty-set identity â€” exactly `seq reduce start (empty {T}) | elem` but without spelling the identity out, so the per-element set type is inferred at type-check. Pure ergonomics: it lowers to a map + a union-fold over existing machinery (no new storage / runtime / engine path). Implemented as a thatable infix binary like `filter`, with a new `Rt::UnionMap` runtime. Adopted by the agent-swarm graph traversal, where it collapses the per-hop frontier expansion to one line. (#221, 2026-06-24)

### Fixed
- **Data race on a repo's in-memory update list â€” `AppendUpdate` vs `GetLowestUpdate`** (#237). The Tetris-merge peek (`GetLowestUpdate`) walked a repo's memtable update ordered-list while holding no lock, but a concurrent client write committing through `~TPusher â†’ AppendUpdate` mutates that same linkage under `DataLock` â€” a real data race on the list (the merge fiber, or another session's peek, vs the writer). It exists on master independent of any flag; the commutative fast-lane (#234) amplifies it by promoting many children per round, so it had to be fixed before the fast-lane could default on. `GetLowestUpdate` now holds `DataLock` across the whole walk (a tag-dispatched `TView` ctor snapshots under the already-held lock to avoid re-locking the non-recursive mutex; no walker machinery re-acquires `DataLock`, and `~TView`/`AcquireCurrentMapping` take only `MappingLock`, preserving the established `DataLock â†’ MappingLock` order, so it cannot deadlock). Verified by re-running the TSan-instrumented `orlyi` under the concurrent-writer bench: the `GetLowestUpdate` race site no longer appears. (closes #237, 2026-06-29)
- **O(NÂ²) writes, and the dormant-merge bug cascade it uncovered** (#227). Writing N keys to a POV was O(NÂ²): the memâ†’disk merge was enqueued only on the emptyâ†’non-empty edge, so under sustained writes the memory layer never drained, grew unbounded, and every write paid an O(layer-size) linear scan. The root cause was an **inverted merge-scheduler deadline** (`deadline < now` where it needed `>`) that meant `StepMergeMem`/`StepMergeDisk` *never ran* â€” so the entire LSM merge / disk-compaction subsystem had been dormant since the 2014-era codebase, and fixing the scheduler exposed a chain of latent correctness and stability bugs that had never been exercised. All fixed here, each gated on write-then-read-back (distinct keys *and* repeated `+=`, single- and multi-POV): **(1)** the scheduler deadline itself, so merges run and the per-write cost is flat (~2 ms/write, measured flat to 8 000 writes); **(2)** the disk compaction fold discarded its computed mutator â€” `TUpdate::NewUpdate`'s map constructor force-tags every entry `Assign`, so a commutative key compacted across several files became `Assign(partial)` and the latest-assign-wins read fold dropped all but one partial (a counter incremented N times read back as a fraction); now folded `Add`/`Or`/â€¦ entries keep their mutator and stay summable; **(3)** the mem-layer consolidation and **(4)** the `ReleasedUpTo` publish had the same mutator-drop / a child-drop race; **(5)** only the root POV now flushes to disk (a child's zero-`UpdateId` disk copy otherwise escaped the cross-repo dedup and double-counted), and the disk present-walker replays a key's history so a merged-to-disk counter reads its full value; **(6)** every WebSocket call leaked one fiber frame (`TJumpRunnable::Main` never freed it), capping an instance at ~`max_parallel_frames` writes; **(7)** the on-disk update index was sized by *update count* rather than *distinct sequence number*, so compaction and the fold pass reserved disk blocks they never wrote and aborted (`allocated blocks that did not get written to`) at a few thousand distinct keys; **(8)** `~TDiskLayer` let a non-shutdown exception escape a `noexcept` destructor during layer cleanup, calling `std::terminate`; **(9)** the atom-layer value serialization (`TCore::NewState`/`Remap`) had no case for the recursive-variant `SelfRef` back-reference leaf (#115) â€” it threw `TCorrupt` when flushing any stored recursive variant, since that traversal had never run; **(10)** the arena-note comparator asserted non-empty `Tuple`/`Record`, crashing on a stored empty record (e.g. a no-payload variant case). (8)â€“(10) surfaced only via the example demos once the flush ran end-to-end (`recursive-variants`, `grc20-pov`). The benchmark ([`examples/agent-swarm/benchmark.py`](examples/agent-swarm/benchmark.py)) is the perf reproduction; its verdict flips from `GROWING` to `FLAT`. Validated across all eight example demos (write-then-read), the `kit2`/indy-disk unit tests, and `lang_test.py`. (closes #227, 2026-06-24 â†’ 2026-06-25)
- **Python client decouples the per-call recv timeout from the connect timeout** (#224). `connect()` passed a single timeout to `create_connection`, which the `websocket-client` library reuses as the socket timeout for every subsequent `recv()` â€” so a method reply that took longer than the (short, connect-sized) window killed an otherwise-fine call, intermittently redding CI on demos unrelated to a PR's change (e.g. a `grc20-pov` read during the `examples/*` job). The connect handshake keeps its short, retried timeout; calls now get a separate generous `recv_timeout` (default 120 s), since a reply is normally milliseconds and a long recv means a starved server, not a hung one. Retrying the call instead would be unsafe (a resent write could double-apply), so it waits rather than retries. The Go and TS clients block indefinitely on a reply and never had this timeout, so they are unaffected. (#224, 2026-06-24)
- **Client `connect()` retries with exponential backoff.** A just-started or heavily loaded `orlyi` can briefly refuse the connection or time out the WebSocket handshake before it is ready, which intermittently flaked the example drivers in CI (a `WebSocketTimeoutException` on connect). All three clients (`clients/python`, `clients/go`, `clients/ts`) now retry a failed connection up to 5 times with exponential backoff (250 ms, doubling), re-raising the last error if it never comes up â€” so every demo's `connect()` (which all delegate to the client libraries) is de-flaked at once. (2026-06-24)
- **Data-race hardening â€” the curated TSan set is now clean and the job gates** (#184). Triaged and fixed every ThreadSanitizer finding the #177 job surfaced, then ratcheted the job from non-gating to gating: (1) the test harness's `TRunner::Pass` flag, written from worker threads, is now `std::atomic` â€” de-noised every multi-threaded test (#190, finding 5); (2) `TVolume`/`TStripedStrategy` background workers now stop+join before teardown, ending a vptr/`close`/`free` dtor race that also crashed the test (#191, findings 2â€“4); (3) the fiber scheduler's manual `ucontext` stack switches are annotated for TSan so it stops mis-attributing fibers and no longer aborts with `longjmp causes uninitialized stack frame` (#194, finding 1); (4) the lock-free fiber work-queue heads, `RunnerArray` publication, and `DebugIsRunning` are `std::atomic` with explicit acquire/release (#199); (5) the `AtomicUnorderedList` Insert/Remove races were racy pre-lock `_mm_prefetch` hints, since removed (#197). (closes #184, 2026-06-22 â†’ 2026-06-23)
- **Atomic checkpoint load.** `TService::LoadCheckpoint` committed the checkpoint's DB data and then installed its packages as two separate steps; if the install threw (missing / bad `.so`, downgrade) the database was left holding data whose packages were never installed. ### Changed
- **Commutative writes no longer read the current value.** A defer-safe commutative `*<[k]>::(T) OP= v` mutation compiled its LHS to `ReadOrIdentity`, which did a full flux read (`TContext::operator[]`) â€” but the codegen consumes only `.GetAddr()`, the value was always discarded, and an absent key already auto-initialises from the monoid identity at fold time, so the read was pure overhead. Since `TContext::operator[]` is O(pending-writes), a transaction of N commutative writes was **O(NÂ²)**, and even a single write paid the cost. The codegen now emits a read-free `IdentityAddr` for the commutative-upsert LHS: **single writes are ~2Ã— faster (3.6 â†’ 1.8 ms under `--mem_sim`)** and a **batched 200-edge transaction goes 39 â†’ 3.1 ms/edge (~12Ã—), with scaling O(NÂ²) â†’ ~flat O(N)** â€” which unblocks bulk load. It also removes a spurious read-dependency on a coordination-free write (#49). (2026-06-26)
- **Committing a large commutative transaction is no longer O(NÂ²).** With the per-write read gone (above), the dominant cost of a big batched write moved to the commit: `TSession::Try` registers each defer-safe commutative mutation with `TUpdate::AddEntry`, which `ReverseInsert`s it into the update's `TKey`-ordered entry collection. The deferred entries arrive in *write* order, so across several indices each insert scanned O(N) to find its slot â€” making a single transaction of N commutative writes **O(NÂ²)** to commit. The entries are now sorted by `TKey` before the `AddEntry` loop, so each insert appends in O(1) (O(N log N) total); correctness is unchanged because `ReverseInsert` always finds the right slot regardless of order â€” only the scan length depended on it. A single-transaction bulk load now commits in **~flat O(N)**: a fresh 2 000-edge batch is ~0.33 ms/edge, holding constant from 200 to 2 000 edges (previously quadratic). This is the second and last O(NÂ²) on the batched-write path; together with the read-elimination above, single-transaction bulk load is now linear end-to-end. (2026-06-26)
