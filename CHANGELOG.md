# Changelog

All notable changes to Orly.

The project does not currently cut numbered releases. Entries reference the originating pull request and the closed issue where applicable. Format loosely follows [Keep a Changelog](https://keepachangelog.com/en/1.1.0/).

## [Unreleased]

### Fixed
- **Added**: `clients/mcp` (`orly-mcp`) — an MCP (Model Context Protocol) server that exposes a running `orlyi` to any MCP-speaking agent runtime as conflict-free shared memory: agents exchange a POV id and write concurrently through `orly_new_pov` / `orly_call` / `orly_call_batch` / `orly_install_package`, with commutative writes merging in the engine. A thin stdio wrapper over the TypeScript driver (JSON→orlyscript escape objects `$set`/`$real`/`$raw` for the shapes JSON lacks); smoke-tested end-to-end in CI against a live server (#526).
- **Fixed**: a gracefully stopped `orlyi` no longer aborts (`SIGABRT`, exit 134) in static destructors after logging `TServer::Shutdown() complete`. The static disk-event pool manager was destroyed in `__run_exit_handlers` while still owning per-thread pools that scheduler-worker and runner threads cannot return without thread-exit hooks; it is now intentionally released at shutdown, like the frame-pool manager already was, and the graceful-shutdown replication fixture now asserts a zero exit status (#522).
- **Fixed**: debug builds no longer abort (`~TRepo` assert, and release builds no longer touch a freed TetrisManager) when an expired or torn-down pov's repo is discarded with updates still parked in its memory layer — e.g. a ttl=0 pov whose owner disconnects during a replication stall. The repo manager now marks such destructions as sanctioned discards; the lifecycle assert still guards every other path, and the dropped-updates case is syslogged (#521).
- **RPC/replication**: an RPC `Write()` racing the connection reader's collapse now returns an immediately-failed future instead of registering one that nothing can ever complete, which parked the master's replicate loop in `Sync()` forever and hung graceful shutdown (#520). The replicate loop's push log is now unconditional and paired with an `acked` line, making a parked-on-unresponsive-peer push observable; the graceful-shutdown fixture waits for an outstanding (written, never acked) push instead of a "fresh" one -- a push racing the slave freeze itself parks the loop before any fresh marker can exist, which is what made CI red (#520) -- keeps its pov out of the layer cleaner's reach (#521), and reports the slave's process state so a dead peer is distinguishable from a frozen one. `jhm --test` flushes its `TEST:`/`EXITCODE:` markers around a failing test's echoed output so CI logs show the failure where it belongs.
- **Graceful shutdown now drains established client connections before clearing the durable cache** (#460). `TServer::Shutdown()` only closed the listening sockets, so a connection still serving (or just idling) held its session's durable pointer into `Durable::TManager::Clear()`, which logged and leaked the object (release) or asserted (debug). Shutdown now hard-closes every established connection's socket right after the listeners go down — the serving loops wind down alongside the existing settle sleep — and waits (bounded, 10s) for the connection objects to actually release their session pointers before the teardown jumper runs `Clear()`. A handshake racing shutdown is refused instead of minting a fresh session behind the drain's back. The graceful-shutdown fixture now holds a handshaked-but-silent client connection across the SIGINT and asserts `Clear()` finds nothing to leak.
- **A pre-restart POV id is now refused cleanly instead of silently resurrecting empty** (#439). The repo save side (`TManager::OnClose`) was never ported, so a POV's un-promoted updates cannot survive a restart — but the POV's *durable record* does, and `TPov::GetRepo` blindly re-created the backing repo, minting an empty shell parented to global whose reads fell through as if the lost writes never happened. A reloaded POV (stream-constructed from a durable disk layer) now only *attaches* to a live repo — which exists on a promoted slave, where replication built it — via the new `TryGetLiveRepo` (live-only lookup, never constructs, never touches the unported #173 reload path); if the repo is gone, the client gets a "povs are ephemeral" error telling it to create a new POV. Documented in PROTOCOL.md; `tests/restart_test.sh` now carries a pre-restart POV id across the restart and asserts the refusal.
- **Graceful shutdown no longer hangs on an unresponsive replication peer** (#461). A master-mode replicate loop parked in `future->Sync()` on a slave RPC was un-wakeable by the stop semaphores: `StopReplicationServices()` now hard-closes the replication socket (`shutdown(SHUT_RDWR)`), which errors the reader loop out of its blocked `Read()`, fails every outstanding RPC future (waking the parked `Sync()`), and lets `JoinReplicationServices()` return. The collapsed reader also skips the failover transition when the death was shutdown's own doing — `Demote()`/`PromoteSlave()` would re-arm `WaitForSlave()` mid-teardown. A new `import_replication.test` fixture pins the scenario: SIGSTOP the slave, park a replication push against the frozen peer, then require a graceful master SIGINT to complete on deadline.
- **The dead `checkpoint` and `command` grammar roots are deleted** (#515). Each language root in `orly.nycr` generates its own nearly-complete ~430KB CST — the whole expression node family, three times over — and two of the three roots have had no live consumers since the flux-capacitor deletion (#262): nothing could feed a checkpoint file or a bare command to anything. Both are gone, along with the checkpoint statement productions, the `imports` keyword they alone used, the checkpoint-only `expr?` postfix production, `new_expr.cc`'s special-case rejection of it, and `cst_utils.h`'s checkpoint walk traits. The two tests that used the dead roots as parse harnesses (`syntax.test`, `reentrant.error.test`) are re-harnessed onto the live package root, their 68 expected trees regenerated from the live parser. One landmine defused en route: `id_literal` looked checkpoint-only at the production level but is a member of the shared `literal` token family — uuid literals are a live expression feature — and with it deleted, `x = {uuid};` silently parsed as a set constructor around a subtraction chain; it stays, and the `IdLiteral` fixture now pins exactly that distinction.
- **Tetris-manager cleanups** (#376). The ad-hoc `throw std::runtime_error("We don't currently support collapsed updates.")` in the global-pov promotion path is now a typed `TCollapsedUpdateError` beside its sibling `TUnsupportedAssertion`. The other half of the issue — "paused frames should join a queue that is re-activated when the semaphore is pushed rather than the current stubbed yield/pop" — turned out to be already true: the pause/unpause handshake runs on `Fiber::TSafeSync`, which parks the waiting frame and re-activates it on `Complete()`, i.e. exactly the mechanism the 2014 stub wished for; what remained was archaeology, so the fossilized `TEventSemaphore`/yield comment blocks are gone and the live mechanism is documented at the site.
- **The "common Io library" wish is dispositioned** (#364). The 2014 TODO wanted `<base/strm/fd.h>` to own `TServeClientRunnable`'s Fd as the first step of making strm the server's I/O layer. Declined, with the rationale at the site: the Fd is already RAII-owned where it lives, the RPC path it feeds is built on `base/io`'s buffered streams with every driver's wire behavior sitting on that stack — migrating it is a protocol-risking rewrite, not a refactor — and twelve years of coexistence have settled the two stream stacks into clear roles (`base/io` under `rpc/`, `base/strm` for the newer flat binary paths like `csv_to_bin` and the importer's file reads).
- **The multiple-tetris-schedulers wish is dispositioned** (#372). Declined, with the rationale at the site: tetris players are already independent fibers multiplexed on the one scheduler runner, the measured tetris bottleneck was the global pov's *sequential* fold — addressed by the #49 commutative fastlane, which sharding runners cannot help — and generalizing to N runners would grow a config surface whose non-default path nothing in CI would exercise while multiplying the teardown join topology the shutdown work just rebuilt. If a many-pov workload ever surfaces a scheduling bottleneck, the shape is: a runner vector, players assigned by parent-pov hash, and the runner budget summing the count.
- **The pump-on-memfd wish is dispositioned** (#288). The 2014 TODO was waiting for `memfd_create` to land; it landed (kernel 3.17) and does not do what the TODO hoped: it makes anonymous *files* — an fd at EOF returns EOF instead of blocking for more data — so it cannot give a consumer stream semantics. What the pump exists to provide, an unbounded never-blocking poll-readable *stream*, is a primitive the kernel still does not have; the pipe pair with a user-space block buffer and a service thread in between is the mechanism, not a workaround. Both sites now say so.
- **`orly/syntax.test` is alive again: 68 CST-shape fixtures pin the grammar** (#504). The file sat disabled (`.broken.cc`) since 2014 with expectations from a grammar that has since moved. Revived mechanically: every expected tree is regenerated from the live parser via the generated CST `Write` dumper (so the fixtures pin the *current* grammar — operator-precedence shapes above all — against future regression), kv-entry inputs gained the id literal the modern `kv_entry` production requires, four fixtures whose 2014 constructs no longer exist were rewritten onto their nearest live equivalents (`free`/`given` onto the `::(type)` spelling, `read ... from` onto the prefix-star read, given-with-default onto given-of-optional), and the two empty shift/rotate placeholder fixtures are deleted — the language never grew those operators, so there is nothing to pin. En route it emerged that the checkpoint *parser* (the grammar root these fixtures use as a harness) has no live consumers since the flux-capacitor deletion (#262); the fixtures' value is the shared expression grammar, which is very much alive.
- **The three disabled tests are live again** (#282). The nondeterministic `Interruption` fixture in `base/util/error.test.cc` was already re-landed deterministically (sigsuspend instead of the racy pause()); the two stragglers: `base/path.test.broken.cc` targeted the pre-simplification TName/TAbsPath API, so it is replaced by a fresh `base/path.test.cc` pinning the current `TPath` surface (parsing shapes including the empty-last-extension executable convention, piece constructors, validity throws, `EndsWith`, printing, the Add/Drop/SwapExtension helpers, and the namespace split/write helpers); the `PeopledirHistogram` walker fixture is rewritten from its 2014 draft into valid orlyscript (`keys` takes the value type and an `@` before the address pattern, sequences come from `reduce` or the `**` unroll) and pins that the walker collects exactly the one object type in the tree. A fourth disabled file surfaced en route — `orly/syntax.test.broken.cc`, 70 CST-shape fixtures against the 2014 grammar — and is session-sized on its own: filed as #504.
- **`TKeyGenerator::TCursor` is a first-class cursor, not a hand-copy of codegen output** (#351). The 2014 comment said it all ("roughly copied from what was generated, as that seemed to work"): a raw owning `TRet*` with a manual `delete` and a hand-nulled move, two duplicated constructors, and dead commented-out initializer lines. Rewritten on the same three-method `Base::TIter` contract with the decoded key in a `std::optional` (validity IS engagement; assignment through an engaged value keeps the storage-reuse behavior), a defaulted move, one constructor, and an advance that asserts on an exhausted underlying cursor instead of walking past it. Lazy decode-once-per-position semantics unchanged. Template-member bodies verified by explicit instantiation probe; the `keys` walk it powers is exercised end-to-end by the lang tests.
- **Package-manager lifecycle hardening: lock-free readers, checked `dlclose`** (#356). The 2014 wish ("engineer the lock out of existence as much as possible") lands as an RCU-style read path: the installed-package map is an immutable snapshot behind a `std::atomic<std::shared_ptr<const TInstalled>>` — the hot per-query `Get()` (and `YieldInstalled`, whose user callback no longer runs under any lock) atomically loads and walks its own pinned snapshot with no lock at all, while the rare writers (`Install`/`Load`/`Uninstall`) serialize on a plain mutex, build the next map aside, and publish it with one atomic store. Semantics preserved: batch error collection, downgrade refusal, same-version no-op, the pre-install callback ordering. The other half of the issue — "we assume dlclose never fails" — is now a checked call in both the destructor and the constructor's error path, logging `dlerror()` instead of silently ignoring a failed unload. The dead `SnapshotInstalled`/`RestoreInstalled` rollback API is deleted: its one consumer (`TService::LoadCheckpoint`, from the #175 checkpoint machinery) went down with the flux-capacitor engine (#262). New `orly/package/manager.test.cc` pins the lifecycle rules and hammers concurrent readers against a churning writer.
- **The generator TODOs are dispositioned, not left to rot** (#360). Both 2014 wishes declined with rationale now documented at the site: the per-`NewCursor` heap allocation is one allocation per generator *layer* per sequence *evaluation* — not per element — and no profile has ever surfaced it (an in-place/value-cursor rework would ripple through every generator, `key_generator.h`, and the generated packages' emissions to shave it); the Zip/Product/Chain/Mix combinators have no orlyscript syntax reaching them, so they would be dead runtime code, and growing the language a surface for them is a design decision, not a backlog item. Riding along: `TFilterGenerator`/`TMapGenerator` cursor copy constructors that copied a `NO_COPY` `TIterHolder` — ill-formed if ever instantiated, alive only because template members instantiate lazily — are now explicitly deleted, and `TIterHolder`'s stale "basically a shared-ptr" comment tells the truth (it owns a `unique_ptr`).
- **The bulk-import / master-slave replication arc** (#367, #497, #498, #499, #500, #501, #503; one PR, pinned end-to-end by the new `orly/server/import_replication.test`, the first master/slave replication coverage anything in CI runs). Writing that test peeled seven stacked defects off the import+replication path, most of them 2014-vintage and latent because the destination block layout, not the code path, decided whether they fired:
  - **#367 — an index id minted by the bulk importer never replicated.** The importer's `SaveIndexNamespaceMapping` + `IndexIdSet` insert lacked the `TIndexIdReplication` enqueue the package-install site has, so a live slave never learned the mapping while ordinary writes to that index still replicated under the raw id — a promoted slave would bind the package to a divergent id and serve empty reads, or abort at startup on the unmapped data files. The duplicated registration logic is now one `TServer::RegisterAndReplicateIndexId` used by both sites (both `TODO(#367)` tags retired). With #498's solo-import rule the enqueue is belt-and-suspenders on the importer path today (a solo server discards its replication queue; joiners get the mapping from the join-time index-map walk, which the test asserts via the slave's "Replicating index" application log) — but the helper keeps install and import behaviorally identical if live import replication ever lands.
  - **#497 — a single-file import passed uninitialized sequence numbers to `AddFileToRepo`.** `ImportCoreVector`'s `final_saved_low/high/num_keys` were only assigned by the merge phase, and a one-file import skips every merge round: debug builds died on the `HighestSeq >= LowestSeq` assert in `TDiskLayer`, release builds silently registered a garbage sequence range. The load jobs now record each written file's `WriteFile` range and the no-merge path seeds the final range from it.
  - **#498 — an import on a replicating master desynchronized sequence numbers and killed the slave.** The importer writes data files and burns a 10M-per-file `UseSequenceNumbers` reservation entirely outside the replication stream, so a live slave died applying the next replicated write (`ensure_or_discard` assert), and even a slave joining *after* the import died in `SyncInventory` (the reservation left highest-materialized ≠ next-1). Imports are now what they factually are, a solo-server bootstrap operation: `BeginImport` refuses unless the server is solo, `WaitForSlave` holds an accepted join until a running import finishes (one gate covers both directions), and the import returns its reservation's unused tail (`TRepo::ReleaseUnusedSequenceNumbers`) so post-import joins sync a consistent range.
  - **#500 — a slave disconnect killed the master.** `WaitForSlave` never released the listener after its one accept was consumed, so the demote path's re-entry died re-binding the slave port (`EADDRINUSE` escaping a fiber is fatal). The listener is released once consumed, and the bind retries while a previous listener drains instead of killing the server. The test kills a slave outright and asserts the master demotes, stays up, and accepts a replacement slave.
  - **#499 — system-repo commits leaked onto the replication stream, with two distinct failure modes.** The system repo is node-local state: each server records its own repos, installed packages, and index-namespace mappings, and a slave rebuilds them from replication *control items* (`TRepoReplication`, `TIndexIdReplication`) — which is why `NewTransaction(false /* we replicate system changes seperately */)` already existed. But the three system-repo writers (`SaveRepo`, `SaveIndexNamespaceMapping`, `SaveInstalledPackage`) used the replicating default, so every install on a paired master (a) shipped a transaction against the slave's *independently numbered* system repo — the `ensure_or_discard` sequence assert killed the slave whenever its local numbering lagged the master's watermark (a coin-flip race), and (b) on the master, the notification pass decoded the commit's void metadata into a `Server::TMetaRecord` unguarded, threw, and silently skipped the batch's remaining client acks (`RunReplicateTransaction error: [not implemented]`). The three writers now pass `false` like the fourth always did, and the notification-pass decode is guarded like its sibling global-pov branch as defense in depth. The test installs a second package on the *paired* master, which also pins the live-stream side of #367: that package's index id must reach the slave via the `TIndexIdReplication` enqueue (the first package's mapping traveled with the join-time walk instead), the master's log must stay free of notification-pass errors, and a subsequent write's ack must still arrive.
  - **#503 — a server push racing `~TClient` aborted the host process.** `Rpc::TContext::Read` calls `shared_from_this()` when the peer sends a request, and the server pushes notification requests at its own cadence; one arriving while the client's shared count was already zero threw `std::bad_weak_ptr` into the dispatch thread's top-level catch, which `abort()`s — taking the whole embedding process with it (reproduced ~1-in-3 by the new test's connection churn). Destruction is now flagged before the service threads are signalled, and their top-level catches exit the thread instead of aborting when teardown is underway; a live client's exception still aborts as before.
  - **#501 — the data file delivered by join-time file sync was unreadable on the slave.** `TMetaRewriter` (which rewrites a synced file's header for the destination block layout) wrote the *block count* where the reader — and the original `TDataFile` writer — use the count of run-length `(block_id, num_blocks)` pairings. Whenever the destination blocks were contiguous (always, on a fresh volume or `--mem_sim`) the reader over-read the pairing list and parsed everything after it as garbage, most visibly the index-segment offsets — the slave's next file merge or read died on a `ByteOffset <= EndOfStream` assert. On 2014's fragmented volumes one-block-per-pairing was common enough for this to ship latent.

  Still open from this arc: the slave/master graceful-shutdown stalls the test routes around with SIGKILL (the #440/#461 teardown cluster owns that ordering).

### Removed
- **The mynde memcached frontend** (closes #352, #368, #369, #370, #371, #374). The `orly/mynde/` protocol library, `TServer`'s memcache surface (`--enable_memcache`/`--memcache_port_number`, the memcache listener/acceptor, `ServeMemcacheClient`, the hard-coded `memcachememcache` index + reserved package name), the Tetris special-cases that replayed mynde `set`s without a package function, and `docs/memcached.md`. It was off by default, exercised by nothing but its own test, spoke a binary protocol memcached itself has deprecated, and the any-language role it was built for in 2014 is filled by the WebSocket frontend. Its removal also let `TSession::AddPov` go private (the memcache path was the one outside caller) and closed the whole open memcache work cluster as moot.
- **The SPA flux_capacitor engine** (#262 P5, ~10K LOC). With `orlyc` running `test{}` blocks on indy (P4 above), nothing consumed the 2014-era SPA engine any longer, so it's deleted: the `orly/spa/` tree (`flux_capacitor/`, the `TService`/`TSpaContext`, honcho, checkpoints, the `spa` binary) and the SPA-only consumers (`orly/manual.cc`, `orly/exit.test.cc`, the SPA-coupled `*.test.broken.cc`). The `--test-engine` flag is gone (indy is the only engine). Three small core untanglings made the deletion clean: `session.cc` uses `Package::TArgMap` instead of `Spa::TArgs::TOrlyArg`; `context_base.h` drops a stale `flux_capacitor/kv.h` include (and pulls `var/sabot_to_var.h` directly); `package/rt.h` drops the unused `TSpaContext` + its `flux_capacitor/api.h` include. Orly now has **one** storage engine end to end — `orlyc` and `orlyi` share indy — and the flux_capacitor POV/Tetris concurrency model (and its footgun class, e.g. #259) is gone. Docs updated (`README.md`, `docs/architecture.md`).

### Fixed
- **server: updates that never touch `now` no longer record run_time = 2013-10-23 in their meta record** (#494). All three commit paths in `session.cc` defaulted the `TMetaRecord` `run_time` with a hardcoded 2013 sentinel (the moment the line was first written) when `GetOptNow()` was unknown. Since `Rt::TContext::Now()` memoizes wall time on first use and the meta record is built after the method runs, the sentinel only landed when neither the client nor the executed method (nor a predicate) ever evaluated `now` — so assertion replay was never affected, but every time-agnostic update was durably recorded as having run on 2013-10-23 17:47:14. The shared `GetRunTime` helper now falls back to the commit-time wall clock, which replay cannot observe (a replayed predicate that reads `now` implies the original evaluated it, forcing the recorded-time path).
- **indy: the on-disk update index's per-sequence-number Metadata/Id cores survive writes and merges** (#323). Three defects in the same seq-index field pair, one of them the issue's original subject: (1) `TMergeDataFile` wrote default-constructed (zeroed) Metadata/Id cores for *every* merged sequence number — the `#if 0` block #323 tracked — so any update surviving an LSM compaction lost its meta record; the merge now streams each source file's update index, keeps the surviving sequence numbers' meta/id notes through the tail-merge keeper filter, remaps their cores into the merged main arena (via a new miss-tolerant `ResolveRemapTolerant` join), and the update-index writer consumes the result in sequence order. (2) `TDataFile` emplaced the cores into its update collector *without* remapping them from the update's in-memory suprena — whose "offsets" are raw note pointers — to the on-disk main arena, so any indirect metadata (every real session update carries a `TMetaRecord`) hit disk as a dangling pointer-valued offset; the notes were always copied into the main arena, only the core offsets were wrong. (3) Both writers seeded their write loop's meta/id from default-constructed cores and only reloaded on a sequence-number change, zeroing the first sequence number's fields even where the rest were right. Read side unchanged (`TUpdateWalkFile` — feeding Tetris's `GetLowestUpdate` promotion and master→slave replication, which is how the zeroed cores were reachable in production). Merging files written before this fix is tolerated: their dangling offsets resolve to nothing and are dropped with a syslog warning, matching their prior (unreadable) state. The four `TODO(#323)` `logic_error` stubs (`TMyMergeArena`/`TMergeIndexFile` `GetStartingBlock`/`ReadMeta`) are unreachable by design — those wrappers' streams locate blocks via `FindPageIdOfByte` — and now say so instead of claiming to be unimplemented. New `MetaIdRoundTrip` fixture pins meta/id round-trips through plain data files (including the first sequence number), a plain merge, and a tail merge that drops a history-only sequence number.
- **base/server: scheduler jobs are cancellable; every long-lived host job is cancel-or-joined at teardown** (#462, the #440/#463 arc's last structural follow-on). `TScheduler::Schedule` queues past the worker cap but never launches another worker, so a long-lived job that never got a worker could neither be joined (its started/exited handshake never ran) nor safely ignored (a worker freeing up mid-teardown would start it against dying state) — the teardown joins had to skip never-started loops and carry a documented residual hazard. New `ScheduleCancelable`/`Cancel`: the queue holds the closure through a shared handle, `Cancel` empties it under the scheduler lock, and a worker taking the job empties it the same way, so exactly one side wins and the owner gets an exact answer — `if (!Cancel(handle)) exited_latch.Pop()`. All the teardown-relevant owners convert: `TServer` (the runner-loop hosts, the BG fast runner, the accept loop — which previously had no join at all — and the housekeeper, whose started-flag skip is retired), `~TDurableManager` (which would previously deadlock outright if a host job starved: it popped fiber sems its writer/merger could never push), and `~TFileService`. `TScheduler::Shutdown` now drains the queue (nothing can start once shutdown begins) and its starved-jobs warning reports only live, never-cancelled leftovers — an owner contract violation, not noise. New scheduler.test fixtures pin cancel-before-take, cancel-after-take, and the refused-job null handle.
- **fiber/server: the #463 teardown race — cross-runner frame handoffs no longer live on the runners** (#463). The lang-test flake class (four one-shot SIGSEGVs plus two hangs, always post-test in the teardown window, never reproducing on rerun) traced to `TRunner::Run`'s peer poll: every runner dereferenced every peer `TRunner` (`RunnerArray[i]->QueueArray[me]`) each lap, but runners die at different times during `TServer::Shutdown()` — the durable manager's writer/merger schedulers and tetris's fiber scheduler are members of managers the teardown fiber destroys while a dozen service runners still poll. A peer that loaded the pointer just before the destruction dereferenced freed, teardown-churned heap and could longjmp a phantom frame's garbage `jmp_buf` (the caught core's exact crash), or lose a queued handoff outright (the hang presentation). The handoff slots now live in a `TRunnerCons`-owned all-pairs matrix that outlives every runner it minted an id for, so the channel is always safe to touch and a handoff pushed around a runner's death is drained instead of dropped; the poll loop touches no peer object at all. Two adjacent unjoined-loop windows closed with the established entered/exited-latch handshake: the scheduler-hosted runner loops (`LaunchSlowFiberSched` jobs: replication, cleaners, merges, BG fast) are now reaped at the end of `Shutdown()` before the runner objects can be destroyed, and `~TFileService` waits for its `BGScheduler` host job the way `~TDurableManager` already did.
- **server: WS responses build their JSON tree directly — no render-to-text/re-parse round trip** (#377). The websocket layer produced every value result (`echo`, `try`, `try batch`) by rendering the var to JSON text (`Var::Jsonify`) and immediately re-parsing it (`TJson::Parse`). New `Var::ToJson` builds the `Base::TJson` tree straight from the var with the identical value mapping (including the dict-key convention: string keys verbatim, composite keys as their compact JSON text), and the three sites use it. One deliberate wire improvement rides along: reals keep full double precision instead of being clipped to the text hop's six significant digits (gated by the examples e2e job, which exercises the drivers end-to-end). The issue's other TODO — jsonifying function *types* in `list packages` — is declined with rationale at the site: the orly-syntax type name is already the JSON representation of a type (a plain string) and there is no round-trip there. New `orly/var/jsonify.test` pins tree-equals-old-round-trip across the var kinds plus the precision improvement.
- **code_gen: an addressless mutable as a mutation target gets a targeted error; four 2014 wishes dispositioned** (#302, #304, #306, #308). New `Rt::TMutable::GetAddrChecked()` — mutating through a mutable that never came from a database read now reports "mutation target has no database address" instead of the generic dereferenced-an-unknown throw (#302's first half; its conflict-detecting-initializer-list rework is declined: the simple per-mutation path is correct, and same-key changes already merge through `ctx.AddEffect` → `TChange::Augment`, which throws on incompatible combinations). Declined with rationale at the sites: a generic object ctor/member printer (#304 — each emission is already one `Base::Join` over the member list; further abstraction trades greppable emission for indirection) and generated-include pruning (#306 — compile time is dominated by the irreducible api/rt includes; per-object headers are tiny, so the reachable-closure walk buys nothing measurable). Test backfill (#308): a new interner fixture prints locals through `WriteStart` and pins the emitted shape (one `auto` per local, dependency order, the outer definition naming the inner local); the "verify generated code compiles" wishes in obj/package/builder are dispositioned to the lang_test suite, which compiles every generated package with gcc end-to-end.
- **var: tuples (the runtime address type) construct and cast through TVar; the disabled address test suites are live again** (#384). The issue's premise was stale: the variadic `Rt::TAddr`/`TAddrElem`/`TAddrImpl` the disabled machinery targeted no longer exists — every occurrence in the tree was inside the `#if 0` blocks this issue tracked, and the runtime address type is a plain `std::tuple` with `TDesc<T>` wrapping descending elements. Re-landed accordingly: `TVar(std::tuple<...>)` and `TVar::Addr(tuple)` unroll via `std::apply` (a `TDesc`-wrapped element contributes the descending direction and its unwrapped value), `TVar::TDt<std::tuple<...>>::As` casts back (rewrapping `TDesc` elements, arity-checked), and `Type::TDt` gains the matching `std::tuple` specialization — which `Var::TMutable`'s conversion path needed all along. The 2014 recursive-unroll dead code is deleted rather than ported, and `Var::TAddr`'s redundant parallel `TypeVec` member is gone (the type derives from the element vars on demand). All four disabled test blocks are re-enabled (rewritten from the dead `Rt::TAddr` spelling to tuples where needed), plus the mutables coverage deferred from #329: a valid `Var::TMutable` needs a real address type, which is finally constructible.

- **code_gen: common subexpression elimination is live again, emitted in dependency order** (#297, #298). The per-scope interner has always deduplicated structurally-identical expression nodes, but the CSE hook it fires on a duplicate fetch (`TCodeScope::AddLocal`) was disabled with "sorting out dependency ordering issues": locals were emitted in arrival order, and when a larger expression is CSE'd before one of its own subexpressions is, the subexpression's `auto` definition would land *after* the local referencing it — use-before-def in the generated C++. `WriteStart` now emits locals in dependency order (stable Kahn over the `AppendDependsOn` edges restricted to the scope's own locals, insertion order as the tiebreak, hard error on the structurally-impossible cycle), which makes every dependency graph load-bearing — hence #298 in the same change: `TMatch`/`TSplit` were the only two of ~30 `AppendDependsOn` implementations inserting a child (`Regex`) without recursing into it (already a live, if narrow, bug via child-func definition ordering). All implementations now record children through one `TInline::AppendDependency` helper whose insert-check keeps transitive walks linear over shared DAGs — sharing is exactly what CSE creates — and breaks recursive-call-body cycles globally (retiring `TIfElse`'s bespoke `InDependsOn` guard). Leaves whose emission is already a name or constant (literals, typed leaves, context vars, arg refs) are excluded from CSE. The `that` context (assert/effect subjects) now materializes on its second use, as a local of the scope that *owns* the context rather than whichever nested scope is active (an inline may only become a local of the scope it was built in, or an inner lambda would define a variable an outer scope references). Safety, verified: `if/else` arms build in their own `TInlineScope`s, so CSE cannot hoist a throwing subexpression past its guard; runtime sequences are cursor factories, so a shared generator is safely re-traversable. Cross-function CSE is declined: gcc already dedups pure expressions in the generated C++, and an Orly-level version would need shared-scope hoisting plus argument threading — new architecture for no demonstrated win. New interner tests pin the duplicate-fetch-creates-a-local behavior, the leaf exclusion, and def-before-use emission under inverted arrival order.
- **disk: `TCompletionTrigger`'s wake/destroy handshake closed — three defects in one rewrite** (#452, the residual instances of the #386/PR #449 free-on-wake class, found by inspection during that arc). The trigger is typically stack-owned and destroyed the moment `Wait()` returns, but `Callback()` bumped `NumFinished` and read `Result` *outside* the spinlock — a fast-path waiter could observe the full count, return, and free the trigger while the final completer was still walking into the (freed) lock to write `Result`. Worse, an error scheduled the waiting frame immediately, *without* the full-count check, so the woken waiter threw out of `Wait()` and freed the trigger under the feet of the still-in-flight straggler completions. And once an error was recorded, the outer `switch (Result)` swallowed every later completion entirely — so an error landing *before* the waiter parked left the trailing successes unable to wake it at all: a hang, not just a race. New contract, all under the spinlock with the wake issued from stack copies after the unlock: the completion count alone decides the wake (the waiter runs only after every outstanding completion has arrived, error or not), `Result` is the sticky first failure and is thrown by `Wait()` at the barrier. New `orly/indy/disk/result.test` joins the curated TSan gate: a sem.test-style reused-storage free-on-wake hammer (with an error mixed into every fourth round), plus deterministic fixtures pinning the error-barrier and error-before-park contracts.
- **indy: LSM merge scheduler no longer enqueues provably-useless disk-merge passes** (#325). The three post-install re-enqueue sites (mem-merge, tail, disk-merge) scheduled another pass whenever the new mapping had ≥3 disk layers, even when the pass's own selection scan — an adjacent disk-layer pair with `SuggestGeneration(lhs) <= SuggestGeneration(rhs)` — could not possibly match. - **indy: cache-read wait-queue considered and declined** (#316). - **jhm: the job runner's "wait for work" condition variable could miss its wakeup** (#337). `Queue()` and `~TJobRunner()` notified `HasWork` after modifying state under `ToRunMutex`, but the queue-runner thread waited on a *different*, unrelated mutex (`HasWorkMutex`, which guarded nothing) with no predicate check — the standard condition_variable footgun: a `notify_all()` that lands before a thread's first `wait()` call is silently dropped. - **The last block of every volume was allocatable and writable but unreadable** (#386, the real root cause). `CheckRange()` in the volume manager -- and `TMemoryDevice::ReadImpl`'s assert -- charged the superblock's `SuperBytes` against `Desc.Capacity`, but `Capacity` counts only payload (devices provide the superblock *in addition*: `TMemoryDevice` allocates `SuperBytes + Capacity`; `MakeVolume` takes one physical block off the exposed count, and the write-side assert already used the payload-only bound). - **Batched / pipelined writes** (#253). A batch `try` statement — `try {pov} pkg method [<{...args1}>, <{...args2}>, ...];` — invokes **one** `(pkg, method)` against **N** argument records and folds all N calls into a **single transaction**. Single-write latency (~800 µs) is at the architecture's floor and ~100% server-side (`docs/design/concurrent-merge-throughput.md` §8–§9), so the remaining multiplicative write lever is amortizing the fixed *per-round-trip* cost — parse, pov-open + `TContext` build + func lookup, commit + destruction, and the network round-trip — across many writes by paying each **once** per batch instead of per call. No new engine subsystem: `TSession::TryBatch` runs the method body N times against the same context and reuses the existing #49/#232 deferred-entry fold (`TContext::AddEffect` already Augment-merges same-key commutative effects) to emit one `TUpdate` / one commit / one `TMetaRecord`. Reply is a JSON **array** of the N per-call results. Semantics: all-or-nothing (a throw in any call aborts the batch before commit), snapshot-isolated (no read-your-writes within a batch — a write-coalescing primitive, not a transaction script), commutative ops to the same key fold while non-commutative assigns collapse last-wins. Clients: `call_batch` (python), `CallBatch` (go), `callBatch` (ts). Benchmark via `examples/agent-swarm/benchmark.py --batch N`; the `examples/wikipedia-pageviews` exact-counter self-check (`DEMO_BATCH=N`) gates correctness. Expected ~3–5× on batchable bulk-load / commutative fan-in; strict one-at-a-time latency-bound callers are unaffected (throughput, not latency). RFC: `docs/design/batched-writes.md` (PR #252). - **Commutative global-merge fast-lane** (#234, behind `--tetris_commutative_fastlane`, **default off**). The single-threaded global-POV Tetris merge re-snapshotted *every* waiting child each round but promoted only one (a 2014 placeholder), so draining N queued child POVs was O(N²) re-snapshot. - **Write backpressure on the merge backlog** (#234, `--tetris_backpressure_threshold`, default 50000; 0 disables). When a writer's POV child repo has more than the threshold of un-promoted updates backed up in its memtable, the accept path cooperatively yields its fiber (`Fiber::YieldSlow`) after committing each write until the global merge drains the backlog below the watermark. So when accept outruns promote, sustained writers degrade to a graceful slowdown that paces accept to promote, instead of the memtable growing until `bad_alloc`. A safety net for the genuine deep-backlog regime; harmless and inert under normal load (the threshold is far above the steady-state backlog).

### Fixed
- **O(W²) reduce-over-range reads — a memory-layer point-read seek accelerator** (#257). The read-side mirror of #227. A fold over a key range — `[a..b] reduce 0 + f(.k: that)`, the canonical time-travel / aggregate read — was O(W²) in fold width: each of the W per-element point reads (`TContext::operator[]`) constructed a `TMemoryLayer::TMatchPresentWalker` whose cursor started at the **head** of the layer's `EntryCollection` (an intrusive doubly-linked ordered list) and linearly scanned to the target key, so reading the key at sorted position *h* cost O(*h*) and the fold summed to ~W²/2. Confirmed three ways (black-box exponent → W^2.0; per-lookup scan-depth instrumentation growing 0→W; a `perf` profile dominated by `MatchPrefixState`/`TCore::NewState` called once per scanned entry). Disk layers were already O(1) per seek via `TIndexManager`; only the in-memory layer head-scanned, and every demo runs `--mem_sim`. Fix: an **intrusive append-only skip list** over `EntryCollection` whose express-lane forward pointers live inside each `TUpdate::TEntry` (no allocation, so it is safe on the `NO_THROW` commit path) and whose **level 0 is `EntryCollection` itself** — the upper lanes are a best-effort accelerator, so a concurrent reader that races an in-flight insert and sees a stale lane simply walks a little more of the authoritative level-0 list (never a wrong result, no new lock, same concurrency profile as the existing list). A fully-bound point read (`operator[]`/`Exists`, threaded through as an `exact_point` flag) seeks via the skip list in O(log W) to the head of the key's run; prefix / `keys`-cursor patterns keep the head-scan (the ordering comparison throws on a free-vs-concrete key, so they must). Per-entry lane height is geometric, seeded by the **index-key hash** (not the SeqNum — batched writes share one SeqNum and would collapse the lanes). The fold flips from O(W²) to O(W): a width-800 `views_total` read drops from 4.4 s to 0.11 s (debug), and per-folded-element cost goes from linearly-growing to flat. Writes stay flat (the O(log W) link is negligible against per-write sabot/commit cost). Validated by a parametrized read-scaling harness (exponent → ~W^0.9), the `wikipedia-pageviews` exact-counter self-check and `agent-swarm` (8 concurrent writers + k-hop traversal, read-back exact), `lang_test.py` (156 pass / 2 xfail), the curated ThreadSanitizer set (`context_fold`/`context_xrepo` clean), and a `memory_layer.test` seek fixture. (closes #257, 2026-06-29)
- **Allocation failure on the write-acceptance path sheds instead of crashing the server** (#250). Under memory pressure (a fixed `--mem_sim` pool running dry, or real `malloc` failing) an allocation failure on the commit path could `std::terminate` the whole `orlyi`, rather than degrade gracefully. Pre-commit failures were already graceful — `Rpc::TAnyRequest::WriteReply` wraps the handler so a `bad_alloc` thrown while *building* a transaction unwinds cleanly (the uncommitted `~TPusher` only `delete`s its copy) and is marshaled back to the client as a structured error. The one genuinely fatal path was post-commit and inside a `noexcept` destructor: `~TPusher → TRepo::AppendUpdate` makes exactly one allocating call, `TTetrisManager::Join` (a `std::map::insert` + possible player construction, run on a child repo's first un-promoted update); under pressure its `bad_alloc` escaped the `noexcept` chain → `terminate` (the `pure virtual method called`/abort in the report). `AppendUpdate` now wraps that `Join` in a `try/catch`: the update is already durable in the memory layer (reads see it) before `Join` runs, so on failure it logs and leaves the repo un-joined — the child→parent **promotion**-join is merely *deferred*, and the next `AppendUpdate` to that repo retries it (the join gate moved from a "first update only" flag to `!InTetris`, which reproduces the success-path semantics exactly — join once while there are un-promoted updates, re-join after a `PopLowest` drain — and additionally retries a previously-failed join). As belt-and-suspenders, the three `NO_THROW` commit-path destructors (`~TPusher`/`~TPopper`/`~TStatusChanger`) now catch-and-log so no allocation ever added to the commit path can silently re-introduce a `terminate`. No proactive pool-watermark shedding was added — a per-`Alloc`/`Free` gauge would regress the hot path, and the backlog-growth→OOM *precursor* is already handled by #234's `--tetris_backpressure_threshold`; this is the orthogonal "an allocation actually failed" mode. Verified by the full `lang_test.py` suite, the concurrent-writer flush-path demos (`wikipedia-pageviews` 8-writer commutative `+=`, `grc20-pov`, `recursive-variants` — all read back exact, promotion intact), and the `transaction_base`/`tetris_manager` unit tests. (closes #250, 2026-06-29)
- **Data race on a repo's in-memory update list — `AppendUpdate` vs `GetLowestUpdate`** (#237). The Tetris-merge peek (`GetLowestUpdate`) walked a repo's memtable update ordered-list while holding no lock, but a concurrent client write committing through `~TPusher → AppendUpdate` mutates that same linkage under `DataLock` — a real data race on the list (the merge fiber, or another session's peek, vs the writer). It exists on master independent of any flag; the commutative fast-lane (#234) amplifies it by promoting many children per round, so it had to be fixed before the fast-lane could default on. `GetLowestUpdate` now holds `DataLock` across the whole walk (a tag-dispatched `TView` ctor snapshots under the already-held lock to avoid re-locking the non-recursive mutex; no walker machinery re-acquires `DataLock`, and `~TView`/`AcquireCurrentMapping` take only `MappingLock`, preserving the established `DataLock → MappingLock` order, so it cannot deadlock). Verified by re-running the TSan-instrumented `orlyi` under the concurrent-writer bench: the `GetLowestUpdate` race site no longer appears. (closes #237, 2026-06-29)
- **O(N²) writes, and the dormant-merge bug cascade it uncovered** (#227). Writing N keys to a POV was O(N²): the mem→disk merge was enqueued only on the empty→non-empty edge, so under sustained writes the memory layer never drained, grew unbounded, and every write paid an O(layer-size) linear scan. The root cause was an **inverted merge-scheduler deadline** (`deadline < now` where it needed `>`) that meant `StepMergeMem`/`StepMergeDisk` *never ran* — so the entire LSM merge / disk-compaction subsystem had been dormant since the 2014-era codebase, and fixing the scheduler exposed a chain of latent correctness and stability bugs that had never been exercised. All fixed here, each gated on write-then-read-back (distinct keys *and* repeated `+=`, single- and multi-POV): **(1)** the scheduler deadline itself, so merges run and the per-write cost is flat (~2 ms/write, measured flat to 8 000 writes); **(2)** the disk compaction fold discarded its computed mutator — `TUpdate::NewUpdate`'s map constructor force-tags every entry `Assign`, so a commutative key compacted across several files became `Assign(partial)` and the latest-assign-wins read fold dropped all but one partial (a counter incremented N times read back as a fraction); now folded `Add`/`Or`/… entries keep their mutator and stay summable; **(3)** the mem-layer consolidation and **(4)** the `ReleasedUpTo` publish had the same mutator-drop / a child-drop race; **(5)** only the root POV now flushes to disk (a child's zero-`UpdateId` disk copy otherwise escaped the cross-repo dedup and double-counted), and the disk present-walker replays a key's history so a merged-to-disk counter reads its full value; **(6)** every WebSocket call leaked one fiber frame (`TJumpRunnable::Main` never freed it), capping an instance at ~`max_parallel_frames` writes; **(7)** the on-disk update index was sized by *update count* rather than *distinct sequence number*, so compaction and the fold pass reserved disk blocks they never wrote and aborted (`allocated blocks that did not get written to`) at a few thousand distinct keys; **(8)** `~TDiskLayer` let a non-shutdown exception escape a `noexcept` destructor during layer cleanup, calling `std::terminate`; **(9)** the atom-layer value serialization (`TCore::NewState`/`Remap`) had no case for the recursive-variant `SelfRef` back-reference leaf (#115) — it threw `TCorrupt` when flushing any stored recursive variant, since that traversal had never run; **(10)** the arena-note comparator asserted non-empty `Tuple`/`Record`, crashing on a stored empty record (e.g. a no-payload variant case). (8)–(10) surfaced only via the example demos once the flush ran end-to-end (`recursive-variants`, `grc20-pov`). The benchmark ([`examples/agent-swarm/benchmark.py`](examples/agent-swarm/benchmark.py)) is the perf reproduction; its verdict flips from `GROWING` to `FLAT`. Validated across all eight example demos (write-then-read), the `kit2`/indy-disk unit tests, and `lang_test.py`. (closes #227, 2026-06-24 → 2026-06-25)
- **Python client decouples the per-call recv timeout from the connect timeout** (#224). `connect()` passed a single timeout to `create_connection`, which the `websocket-client` library reuses as the socket timeout for every subsequent `recv()` — so a method reply that took longer than the (short, connect-sized) window killed an otherwise-fine call, intermittently redding CI on demos unrelated to a PR's change (e.g. a `grc20-pov` read during the `examples/*` job). The connect handshake keeps its short, retried timeout; calls now get a separate generous `recv_timeout` (default 120 s), since a reply is normally milliseconds and a long recv means a starved server, not a hung one. Retrying the call instead would be unsafe (a resent write could double-apply), so it waits rather than retries. The Go and TS clients block indefinitely on a reply and never had this timeout, so they are unaffected. (#224, 2026-06-24)
- **Client `connect()` retries with exponential backoff.** A just-started or heavily loaded `orlyi` can briefly refuse the connection or time out the WebSocket handshake before it is ready, which intermittently flaked the example drivers in CI (a `WebSocketTimeoutException` on connect). All three clients (`clients/python`, `clients/go`, `clients/ts`) now retry a failed connection up to 5 times with exponential backoff (250 ms, doubling), re-raising the last error if it never comes up — so every demo's `connect()` (which all delegate to the client libraries) is de-flaked at once. (2026-06-24)
- **Data-race hardening — the curated TSan set is now clean and the job gates** (#184). ### Changed
- **Commutative writes no longer read the current value.** A defer-safe commutative `*<[k]>::(T) OP= v` mutation compiled its LHS to `ReadOrIdentity`, which did a full flux read (`TContext::operator[]`) — but the codegen consumes only `.GetAddr()`, the value was always discarded, and an absent key already auto-initialises from the monoid identity at fold time, so the read was pure overhead. Since `TContext::operator[]` is O(pending-writes), a transaction of N commutative writes was **O(N²)**, and even a single write paid the cost. The codegen now emits a read-free `IdentityAddr` for the commutative-upsert LHS: **single writes are ~2× faster (3.6 → 1.8 ms under `--mem_sim`)** and a **batched 200-edge transaction goes 39 → 3.1 ms/edge (~12×), with scaling O(N²) → ~flat O(N)** — which unblocks bulk load. It also removes a spurious read-dependency on a coordination-free write (#49). (2026-06-26)
- **Committing a large commutative transaction is no longer O(N²).** With the per-write read gone (above), the dominant cost of a big batched write moved to the commit: `TSession::Try` registers each defer-safe commutative mutation with `TUpdate::AddEntry`, which `ReverseInsert`s it into the update's `TKey`-ordered entry collection. The deferred entries arrive in *write* order, so across several indices each insert scanned O(N) to find its slot — making a single transaction of N commutative writes **O(N²)** to commit. The entries are now sorted by `TKey` before the `AddEntry` loop, so each insert appends in O(1) (O(N log N) total); correctness is unchanged because `ReverseInsert` always finds the right slot regardless of order — only the scan length depended on it. A single-transaction bulk load now commits in **~flat O(N)**: a fresh 2 000-edge batch is ~0.33 ms/edge, holding constant from 200 to 2 000 edges (previously quadratic). This is the second and last O(N²) on the batched-write path; together with the read-elimination above, single-transaction bulk load is now linear end-to-end. (2026-06-26)
