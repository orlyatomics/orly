# Changelog

All notable changes to Orly.

The project does not currently cut numbered releases. Entries reference the originating pull request and the closed issue where applicable. Format loosely follows [Keep a Changelog](https://keepachangelog.com/en/1.1.0/).

## [Unreleased]

### Removed
- **The mynde memcached frontend** (closes #352, #368, #369, #370, #371, #374). The `orly/mynde/` protocol library, `TServer`'s memcache surface (`--enable_memcache`/`--memcache_port_number`, the memcache listener/acceptor, `ServeMemcacheClient`, the hard-coded `memcachememcache` index + reserved package name), the Tetris special-cases that replayed mynde `set`s without a package function, and `docs/memcached.md`. It was off by default, exercised by nothing but its own test, spoke a binary protocol memcached itself has deprecated, and the any-language role it was built for in 2014 is filled by the WebSocket frontend. Its removal also let `TSession::AddPov` go private (the memcache path was the one outside caller) and closed the whole open memcache work cluster as moot.
- **The SPA flux_capacitor engine** (#262 P5, ~10K LOC). With `orlyc` running `test{}` blocks on indy (P4 above), nothing consumed the 2014-era SPA engine any longer, so it's deleted: the `orly/spa/` tree (`flux_capacitor/`, the `TService`/`TSpaContext`, honcho, checkpoints, the `spa` binary) and the SPA-only consumers (`orly/manual.cc`, `orly/exit.test.cc`, the SPA-coupled `*.test.broken.cc`). The `--test-engine` flag is gone (indy is the only engine). Three small core untanglings made the deletion clean: `session.cc` uses `Package::TArgMap` instead of `Spa::TArgs::TOrlyArg`; `context_base.h` drops a stale `flux_capacitor/kv.h` include (and pulls `var/sabot_to_var.h` directly); `package/rt.h` drops the unused `TSpaContext` + its `flux_capacitor/api.h` include. Orly now has **one** storage engine end to end — `orlyc` and `orlyi` share indy — and the flux_capacitor POV/Tetris concurrency model (and its footgun class, e.g. #259) is gone. Docs updated (`README.md`, `docs/architecture.md`).

### Fixed
- **Malformed `mutable` types and mismatched mutation assignments now get real diagnostics** (#314, the two sites left open by the sweep's partial fix). A `mutable (mutable int)` type — or a `mutable @ T ...` whose `@` type is not an address — assertion-failed inside `Type::TMutable::Get`, aborting `orlyc`; the synth type builder now checks those preconditions itself and reports "a mutable type cannot contain another mutable type" / "the '@' type of a mutable must be an address type" at the `mutable` keyword's position. A mutation assignment whose new value's type doesn't match the stored value's (`x <- "nope"` where `x` holds an `int`) said only "This expression is invalid."; it now names both types ("cannot assign a value of type ... to a mutable holding ..."), while the assign path's already-specific diagnostics (optionals in addresses, sequences, object subsets) pass through untouched. Enabled by the `TUnaryType::TGet`/`TBinaryType::TGet` `std::function` generalization, which lets the builder capture the pos-range. Two new diagnostics-pinning lang_tests fail compilation by design and pin the exact messages via their `.state` baselines.
- **jhm: removing a config file now invalidates the cache** (#339). Incremental rebuilds only noticed config *additions* (a new `foo.cc.jhm` bumps the file's effective timestamp); deleting one made the timestamp revert to older-than-the-outputs, so stale flags kept being served from cache. Each job's cache now records the exact set of config files that fed the build (env config search list + input/needs/outputs `*.jhm` files), and the cache check recomputes and compares the set — a recorded file that's gone, or a present file the cache never saw whose mtime predates the outputs, forces a rebuild. Caches written before this key existed rebuild once on first use.
- **`Base::MemAlignedAlloc` freed `posix_memalign` memory with `operator delete`** (PR #465). The helper returned a plain `std::unique_ptr`, so every aligned disk buffer — page cache, mem-sim volume, block/discard maps, buf blocks, block-hit counters — was released with the wrong deallocator: an alloc/dealloc mismatch latent since 2014, because the buffers were process-lifetime and #440's real teardown was the first thing to ever free them. The helper now returns a `free()`-deleting `Base::TMemAlignedPtr<T>` and the six member declarations adopt it. Found by the new `asan-smoke` CI job on its first runs — along with a `TCmd` stack-use-after-scope in orlyc's shutdown-on-every-path flow (#459-introduced, also fixed): `TServer` keeps a reference to its `TCmd`, and the try-scoped `server_cmd` died before `Shutdown()` read it. The smoke now gates the teardown paths under AddressSanitizer on push-to-master and the nightly.
- **`balancer.test` intermittent teardown hang** (#455). `TBalancer` scheduled its accept/serve jobs with no teardown handshake, so a `ServeClient` proxy loop could park in `epoll_wait` forever while the balancer (and the test) tried to shut down. `TBalancer` now runs every job through a shared-ptr'd handshake block — jobs refuse to start once shutdown has begun, watch a term fd in their epoll sets, and are counted in/out — and the new `StopServing()` (called at the top of each concrete destructor, then idempotently by `~TBalancer`) flips the flag, pokes the term fd, and blocks until all in-flight jobs have exited. Same treatment for the test harness itself, where TSan showed the matching bug class: `TTestServer`'s destructor waited only for its accept job while spawned `ServeClient` jobs could still be answering an RPC through the freed server (heap-use-after-free), and the `Running` stop flags were unsynchronized plain bools (now atomic). Also: `TFailoverTestBalancer`'s destructor waits on a predicate instead of a bare `wait` (correct if `CheckHosts` already exited or unwound by exception), the test moved to dedicated ports away from the default Orly service ports (stray local Orly activity was landing foreign connections on the balancer mid-test), and all test-harness socket I/O is timeout-bounded so a lost reply retries instead of freezing the run. balancer.test is now TSan-clean locally (was 3-4 warnings/run on master). Three emitters fixed: control characters below 0x20 without a short escape now come out as `\u00XX` in `TJson::WriteString` (raw multi-byte UTF-8 still passes through, which is legal, and `ReadQuotedString`'s existing `\u` handler round-trips the escapes); a dict with a composite (non-string) key now renders the key's JSON and emits it as a properly escaped string instead of naively wrapping it in quotes (which produced nested unescaped quotes); and non-finite reals (NaN/Infinity, which JSON cannot represent) emit `null`. Matters because the WS protocol surfaces `Jsonify` to every client -- and `ws.cc` itself re-parses the output through `TJson::Parse`, so invalid output was a server-side throw path too.
- **A probed device joining an existing volume is now checked against the volume's description** (#319). Every device's superblock carries the volume-level geometry (strategy, speed, replication factor, extent count, striping); previously only the *device*-level desc was verified on `AddDevice`, so a corrupt superblock or mis-assembled device set produced a volume whose geometry silently depended on probe order. `TVolume::TDesc` gains an `operator!=` and `TDiskUtil` refuses the mismatched device with a descriptive error.
- **The last block of every volume was allocatable and writable but unreadable** (#386, the real root cause). `CheckRange()` in the volume manager -- and `TMemoryDevice::ReadImpl`'s assert -- charged the superblock's `SuperBytes` against `Desc.Capacity`, but `Capacity` counts only payload (devices provide the superblock *in addition*: `TMemoryDevice` allocates `SuperBytes + Capacity`; `MakeVolume` takes one physical block off the exposed count, and the write-side assert already used the payload-only bound). So once a volume filled to its tail, the first *read* of the final physical block threw `std::logic_error` -- and when that read was `~TDiskOrderedLayer`'s (the durable layer cleaner, a noexcept destructor), the whole server aborted with `terminate called without an active exception`: the load-sensitive lang_test SIGABRT flake #386 chased across three fixes. Root-caused from 4 identical core dumps plus an instrumented suite run (12/12 hits: a 4K read at `Capacity - SuperBytes`, overshoot 4096 < SuperBytes 65536, on orlyc's 8 MB mem-sim volume). Fixed the bound in both places; hardened `~TDiskOrderedLayer` to log-and-leak instead of dying on any remaining reclaim failure; made `NextDurableByIdGenId` atomic (the writer and merger fibers mint gen ids from it on different threads). New `volume_manager.test` drains the fast volume's allocator and round-trips its device-tail block -- red on the old bound with the exact production exception, green after.
- **Fiber sync-primitive wake/destroy handshake** (#386 residual). `TSingleSem::Push`, `TSem::Push`, and `TSafeSync::Complete` made their waiter runnable from inside the spinlock scope and then released the spinlock — touching the primitive after the waiter could resume, return, and destroy it (the same class #407 fixed for `TJumpRunnable`, and the cause of the remaining load-sensitive lang_test SIGABRT flake). Waiters in `Pop`/`Sync` now take-and-release the spinlock once after waking (the POSIX "a mutex may be destroyed right after unlock" pattern), so the signaller's unlock is provably its last touch; `TSafeSync::Complete` additionally moves its `Finished` increment under the lock, closing the same hole on `Sync()`'s no-wait fast path. New `orly/indy/fiber/sem.test` hammers the free-on-wake idiom for all three primitives — ThreadSanitizer flags the old code deterministically and is clean after — and joins the curated TSan gate along with `jump_runnable.test`. Unblocks the #277 durability-sem reland.
- **Durability signal fired before the data was durable** (#277). `TDurableManager::Save()` pushed the caller's semaphore as soon as the save landed in the in-memory slush layer, so `Durable::TObj`'s save-on-close "wait for durability" was theater and a crash between signal and flush silently lost the save; the destructor additionally dropped whatever was still unflushed. The sem now rides the slush entry and fires only after the sorted-by-id file's disk writes are confirmed; the writer runs a final drain at shutdown (flush, then release any savers) and late saves self-signal. The sem is now nullable — the three fire-and-forget callers (session-create's save-for-replication, both replication-slave loops) passed never-popped stack sems that would otherwise have become use-after-frees. New `durable_manager.test` (this subsystem's first) pins the contract and, under TSan, surfaced + fixed two latent teardown races (non-atomic `ShutDown` flag; `TRunner` members destroyed while their scheduler-job loops could still scan them). (First landed as PR #443, reverted in #445 because its timing shift amplified the then-unfixed #386 sync-primitive residual in CI; relanded unchanged on top of the fix above.)
- **Tetris teardown races** (#280, PR #441). Three defects in the `TTetrisManager`/`TPlayer` stop path, re-measured as real after #407 (6/300 functional failures, 50/50 TSan): `Stop()` waited on a member sync that the self-deleting player's destructor completed (use-after-free; replaced with a stack atomic-flag handshake whose flip is the dying fiber's last act); `StopAllPlayers()` held `FiberMutex` across `Stop()` while a mid-`Play()` player can re-enter the manager through that mutex (deadlock; the map is now snatched under the lock and stopped outside it, with a `Stopping` guard and a live-player count so even self-destructing players are gone before povs are torn down); and the unit test's own drain-announcement race caused the intermittent `48 != 72` lost-update CI failures (drains now confirmed only after the onward push). The teardown waits are fiber-context-agnostic, unblocking the #440 `TServer::Shutdown()` plan. The residual `lists_of_*` SIGABRT flake was re-confirmed and #386 reopened with leads. (The companion #277 durability-sem fix, PR #443, was merged and then **reverted**: correct under its own gates — TSan-clean, restart-durability green — but its timing shift made the reopened #386 residual fire in CI's lang_test at 3-5 aborts/run, two consecutive reds; it relands once the #386 sync-primitive handoff is fixed.)

- **Fiber completion-handshake race** (#386, PR #407). `TJumpRunnable`'s completion handshake could race its own teardown, killing `orlyc`'s embedded indy test server with a SIGABRT (`terminate called without an active exception`), a silent hang, or a SIGSEGV depending on interleaving — the dominant CI/lang_test flake of the revival period (also suspected behind the #247 examples read-hang and today's `tetris_manager.test` incidents; both flagged for post-fix re-measurement).
- **Latent-bug harvest from the TODO-backlog sweep** (2026-07-01/02). Working the #275–#384 backlog surfaced and fixed six genuine defects, several dating to 2014: `TConverter`'s integer-underflow check was mathematically never true, so one-past-`LONG_MIN` inputs silently wrapped (found by new #334 tests; the tripping digit was also folded in as signed-overflow UB); web-counter sampling dropped concurrent increments (read-then-zero race → atomic `exchange`, #292); generated objects hashed members with bare XOR, so any two equal members cancelled to zero (#295); the `user_id` context variable emitted a runtime call that doesn't exist, making the surface uncompilable since 2014 (#310); `TWith`'s copied child-function emission never got `TFunction`'s dependency ordering, so mutually-referencing helpers in a `with` block could generate uncompilable C++ (#311); and `Rt::Match`/`MatchLess` over tuples threw a placeholder on the code-generated comparison path (#362). Also: duplicate literal dict keys are now a compile error naming the key (#378), and `csv_to_bin` scans doubles in place instead of slurping the field (#313).

### Added
- **Orderly `TServer` shutdown — construct/run/destroy is a first-class lifecycle** (#440 arc: slice A PR #442, slice B PR #447, slice C PR #459; the design record is `docs/teardown-design.md`). `TServer::Shutdown()` stops the world in dependency order: cut the work sources (listeners, ws), let the in-flight pipeline settle on its own cadence, then stop **and join** every service loop — a stop is only a flag plus a wake, so each loop confirms its exit via a started-counter/exited-latch handshake (exception-safe) before the managers hosting its state are destroyed — flush every dirty repo memory layer and the durable slush inline (a graceful stop loses nothing; closes #440), and finally destroy everything: Tetris, the durable manager (cached sessions/povs destroyed so their repo pins drop), `GlobalRepo`, and the repo manager (`MakeDirty()` self-pins released — paused-and-written test povs are never promoted, so nothing else ever drops them — and `PreDtor` verifies every repo ptr is gone, naming any leaker). `orlyi` wires SIGINT → Shutdown → clean exit; `orlyc` drops its leak-and-`_exit` policy, so every compile's `test{}` run doubles as a construct/run/destroy smoke of the whole server. En route: `TScheduler` now logs jobs that were scheduled but never ran (the silent worker-cap starvation that had quietly disabled the merge runners and housekeeper under orlyc's old 8-worker policy), orlyc's machine-form stdout is kept as a pure compiler protocol (server noise goes to syslog; it broke all 28 baseline-bearing lang tests when the starved jobs first ran), and `lang_test.py`'s file-mode xfail tracking was fixed. Follow-ons tracked: #460 (drain live connections before durable `Clear()`), #461 (a master-mode `Sync()` holds the shutdown join), #462 (queued-job cancellation), #463 (one lang_test exit-code flake). The ASan teardown smoke intended here shipped as PR #465 — no fiber-switch annotations proved necessary (unlike TSan, #194; `-U_FORTIFY_SOURCE` in `asan.jhm` is the one requirement) — and its first runs caught two real teardown bugs (see Fixed).
- **`orlyc` reports its SCM version** (#353). `make version` (a `debug`/`release` prerequisite) generates a gitignored `orly/build_version.h` from `git describe` — write-if-changed via temp+rename, so no spurious rebuilds and no torn header — and `orlyc --help` prints `Build: <version>` instead of the 2014 `Build: Unknown`. Direct `jhm` invocations fall back to `unknown` via `__has_include`.
- **Hex and octal integer parsing in `TConverter`** (#285). `0x`/`0X` and python-style `0o`/`0O` prefixes (a bare leading zero stays decimal), with the same overflow/underflow discipline as decimal, UBSan-verified at the `LONG_MIN`/`LONG_MAX` boundaries. Format strings — the other half of the 2014 wish — declined as consumer-less.
- **Run `test{}` blocks on indy** (#262, behind `orlyc --test-engine=indy`, **default `spa`**). `orlyc` runs a package's compile-time `test{}` blocks to fail the build on a failed assertion; the *only* reason the 2014 SPA flux_capacitor engine (`orly/spa/`, ~10K LOC) still exists is that it was the sole engine that ran them. This finishes the abandoned indy-side test runner (`TSession::RunFuncCommit` / `RunTestBlock` / `RunTestSuite`, previously stubbed) and embeds a short-lived in-process **memory-simulated** `TServer` in `orlyc` (`RunTestsOnIndy` → new public `TServer::RunPackageTests`, run on a fast runner; the server is intentionally leaked + `_exit`'d, exactly as `orlyi` never destroys its own), so the compiler can drive the real engine without a network server. The package code was always engine-neutral (`Runner` takes the abstract `Package::TContext&`), so this is a finish-and-switch, not a rewrite. **Full parity:** the whole `lang_test` suite passes identically on both engines (156 passed / 2 xfail), with no SPA regression and no measurable compile-with-tests latency cost (per-invocation server bring-up is ~0.15 s, lost in the ~15 s gcc compile of the generated package every build already pays). Two engine-semantics points were needed for parity, both confined to the compile-time test path: (1) a fresh test POV inherits its parent's sequence counter (`SeedTestPovSequence`) so the point-read fold prefers descendant writes down the test POV chain; (2) the test commit resolves each mutation against the current value (read-modify-write) rather than using the #49 deferred-commutative path (the read-time fold cannot combine *mixed* mutators across the chain, e.g. `*=` over a `+=`-built value), while an absent key under a defer-safe-commutative op still upserts from the monoid identity (#151). Also: `sched_setaffinity` pinning in the server is now best-effort (log + run unpinned) so affinity-restricted containers/CI don't fail startup, and `lang_test.py` honors `ORLYC` / `ORLYC_EXTRA_ARGS` to run the suite against either engine. Default stays `spa`; flipping the default and deleting the SPA engine are the follow-up phases. RFC: `docs/design/unify-test-execution-on-indy.md` (#263). (#262)
- **Batched / pipelined writes** (#253). A batch `try` statement — `try {pov} pkg method [<{...args1}>, <{...args2}>, ...];` — invokes **one** `(pkg, method)` against **N** argument records and folds all N calls into a **single transaction**. Single-write latency (~800 µs) is at the architecture's floor and ~100% server-side (`docs/design/concurrent-merge-throughput.md` §8–§9), so the remaining multiplicative write lever is amortizing the fixed *per-round-trip* cost — parse, pov-open + `TContext` build + func lookup, commit + destruction, and the network round-trip — across many writes by paying each **once** per batch instead of per call. No new engine subsystem: `TSession::TryBatch` runs the method body N times against the same context and reuses the existing #49/#232 deferred-entry fold (`TContext::AddEffect` already Augment-merges same-key commutative effects) to emit one `TUpdate` / one commit / one `TMetaRecord`. Reply is a JSON **array** of the N per-call results. Semantics: all-or-nothing (a throw in any call aborts the batch before commit), snapshot-isolated (no read-your-writes within a batch — a write-coalescing primitive, not a transaction script), commutative ops to the same key fold while non-commutative assigns collapse last-wins. Clients: `call_batch` (python), `CallBatch` (go), `callBatch` (ts). Benchmark via `examples/agent-swarm/benchmark.py --batch N`; the `examples/wikipedia-pageviews` exact-counter self-check (`DEMO_BATCH=N`) gates correctness. Expected ~3–5× on batchable bulk-load / commutative fan-in; strict one-at-a-time latency-bound callers are unaffected (throughput, not latency). RFC: `docs/design/batched-writes.md` (PR #252). (closes #253)
- **Commutative global-merge fast-lane** (#234, behind `--tetris_commutative_fastlane`, **default off**). The single-threaded global-POV Tetris merge re-snapshotted *every* waiting child each round but promoted only one (a 2014 placeholder), so draining N queued child POVs was O(N²) re-snapshot. The fast-lane promotes **all** assertion-free (commutative `+=`/`|=`) children per round, each in its own transaction, so the one-Pusher-per-repo invariant and each child's session-id metadata (load-bearing for replication notifications) are preserved exactly; assertion-bearing RMW children keep the one-per-round discipline byte-for-byte. New reporter counter `Tetris Children Considered / Push` makes the quadratic observable: it drops from ~30 to 1.0 at 32 child POVs. Shipped default-off (PR #236). **The default-on flip was investigated and declined** (this PR): the §8.6 prerequisites were resolved — the pre-existing race the fast-lane amplifies is fixed (#237, below) and write backpressure was added (below) — but a K=24 release soak (both synchronous and pipelined, on the Tetris path) measured the flag throughput- *and* sustainability-neutral: with it on or off the run degrades identically, because the binding concurrent-write constraint is the per-write acceptance path / fast-scheduler capacity, not the merge tournament (the merge keeps up; backlog ≈1.3 children/round). So the flag stays off, available for the genuine deep-backlog regime. The benchmark (`examples/agent-swarm/concurrent_bench.py`, `run-concurrent-bench.sh`) ships with it. See `docs/design/concurrent-merge-throughput.md` §8.
- **Write backpressure on the merge backlog** (#234, `--tetris_backpressure_threshold`, default 50000; 0 disables). When a writer's POV child repo has more than the threshold of un-promoted updates backed up in its memtable, the accept path cooperatively yields its fiber (`Fiber::YieldSlow`) after committing each write until the global merge drains the backlog below the watermark. So when accept outruns promote, sustained writers degrade to a graceful slowdown that paces accept to promote, instead of the memtable growing until `bad_alloc`. A safety net for the genuine deep-backlog regime; harmless and inert under normal load (the threshold is far above the steady-state backlog).
- **`union_map` sequence operator** (#219 phase 2). `seq union_map elem` maps each element of a sequence to a set and unions the results, from an implicit empty-set identity — exactly `seq reduce start (empty {T}) | elem` but without spelling the identity out, so the per-element set type is inferred at type-check. Pure ergonomics: it lowers to a map + a union-fold over existing machinery (no new storage / runtime / engine path). Implemented as a thatable infix binary like `filter`, with a new `Rt::UnionMap` runtime. Adopted by the agent-swarm graph traversal, where it collapses the per-hop frontier expansion to one line. (#221, 2026-06-24)
- **Frontier-delta BFS for graph traversal** (#219 phase 3). The agent-swarm `reach` no longer re-expands the entire visited set every hop (O(k·|visited|·degree)); it carries a `<{.seen, .frontier}>` accumulator and expands only the new frontier, so each node's adjacency is read exactly once (O(edges)) — what lets traversal scale past toy graphs. Same reachable set as the naive whole-set fold (kept as `reach_naive`; the inline + lang tests assert `reach == reach_naive` hop for hop). Pure orlyscript, no engine change; the 25-node demo result is unchanged. (#223, 2026-06-24)
- **Transitive graph traversal over the agent-swarm knowledge graph** (#219 phase 1). The cooccurrence graph is now traversable: `add_cooccur` writes a symmetric `<['adj', a, b]>` adjacency both ways — a trailing-`free` prefix scan the sorted index seeks straight to (index-free adjacency) — and bounded k-hop reachability falls out of existing pieces with **no new engine primitive**: a `reduce` over a depth range accumulating a visited set (so cycles terminate). ### Fixed
- **O(W²) reduce-over-range reads — a memory-layer point-read seek accelerator** (#257). The read-side mirror of #227. A fold over a key range — `[a..b] reduce 0 + f(.k: that)`, the canonical time-travel / aggregate read — was O(W²) in fold width: each of the W per-element point reads (`TContext::operator[]`) constructed a `TMemoryLayer::TMatchPresentWalker` whose cursor started at the **head** of the layer's `EntryCollection` (an intrusive doubly-linked ordered list) and linearly scanned to the target key, so reading the key at sorted position *h* cost O(*h*) and the fold summed to ~W²/2. Confirmed three ways (black-box exponent → W^2.0; per-lookup scan-depth instrumentation growing 0→W; a `perf` profile dominated by `MatchPrefixState`/`TCore::NewState` called once per scanned entry). Disk layers were already O(1) per seek via `TIndexManager`; only the in-memory layer head-scanned, and every demo runs `--mem_sim`. Fix: an **intrusive append-only skip list** over `EntryCollection` whose express-lane forward pointers live inside each `TUpdate::TEntry` (no allocation, so it is safe on the `NO_THROW` commit path) and whose **level 0 is `EntryCollection` itself** — the upper lanes are a best-effort accelerator, so a concurrent reader that races an in-flight insert and sees a stale lane simply walks a little more of the authoritative level-0 list (never a wrong result, no new lock, same concurrency profile as the existing list). A fully-bound point read (`operator[]`/`Exists`, threaded through as an `exact_point` flag) seeks via the skip list in O(log W) to the head of the key's run; prefix / `keys`-cursor patterns keep the head-scan (the ordering comparison throws on a free-vs-concrete key, so they must). Per-entry lane height is geometric, seeded by the **index-key hash** (not the SeqNum — batched writes share one SeqNum and would collapse the lanes). The fold flips from O(W²) to O(W): a width-800 `views_total` read drops from 4.4 s to 0.11 s (debug), and per-folded-element cost goes from linearly-growing to flat. Writes stay flat (the O(log W) link is negligible against per-write sabot/commit cost). Validated by a parametrized read-scaling harness (exponent → ~W^0.9), the `wikipedia-pageviews` exact-counter self-check and `agent-swarm` (8 concurrent writers + k-hop traversal, read-back exact), `lang_test.py` (156 pass / 2 xfail), the curated ThreadSanitizer set (`context_fold`/`context_xrepo` clean), and a `memory_layer.test` seek fixture. (closes #257, 2026-06-29)
- **Allocation failure on the write-acceptance path sheds instead of crashing the server** (#250). Under memory pressure (a fixed `--mem_sim` pool running dry, or real `malloc` failing) an allocation failure on the commit path could `std::terminate` the whole `orlyi`, rather than degrade gracefully. Pre-commit failures were already graceful — `Rpc::TAnyRequest::WriteReply` wraps the handler so a `bad_alloc` thrown while *building* a transaction unwinds cleanly (the uncommitted `~TPusher` only `delete`s its copy) and is marshaled back to the client as a structured error. The one genuinely fatal path was post-commit and inside a `noexcept` destructor: `~TPusher → TRepo::AppendUpdate` makes exactly one allocating call, `TTetrisManager::Join` (a `std::map::insert` + possible player construction, run on a child repo's first un-promoted update); under pressure its `bad_alloc` escaped the `noexcept` chain → `terminate` (the `pure virtual method called`/abort in the report). `AppendUpdate` now wraps that `Join` in a `try/catch`: the update is already durable in the memory layer (reads see it) before `Join` runs, so on failure it logs and leaves the repo un-joined — the child→parent **promotion**-join is merely *deferred*, and the next `AppendUpdate` to that repo retries it (the join gate moved from a "first update only" flag to `!InTetris`, which reproduces the success-path semantics exactly — join once while there are un-promoted updates, re-join after a `PopLowest` drain — and additionally retries a previously-failed join). As belt-and-suspenders, the three `NO_THROW` commit-path destructors (`~TPusher`/`~TPopper`/`~TStatusChanger`) now catch-and-log so no allocation ever added to the commit path can silently re-introduce a `terminate`. No proactive pool-watermark shedding was added — a per-`Alloc`/`Free` gauge would regress the hot path, and the backlog-growth→OOM *precursor* is already handled by #234's `--tetris_backpressure_threshold`; this is the orthogonal "an allocation actually failed" mode. Verified by the full `lang_test.py` suite, the concurrent-writer flush-path demos (`wikipedia-pageviews` 8-writer commutative `+=`, `grc20-pov`, `recursive-variants` — all read back exact, promotion intact), and the `transaction_base`/`tetris_manager` unit tests. (closes #250, 2026-06-29)
- **Data race on a repo's in-memory update list — `AppendUpdate` vs `GetLowestUpdate`** (#237). The Tetris-merge peek (`GetLowestUpdate`) walked a repo's memtable update ordered-list while holding no lock, but a concurrent client write committing through `~TPusher → AppendUpdate` mutates that same linkage under `DataLock` — a real data race on the list (the merge fiber, or another session's peek, vs the writer). It exists on master independent of any flag; the commutative fast-lane (#234) amplifies it by promoting many children per round, so it had to be fixed before the fast-lane could default on. `GetLowestUpdate` now holds `DataLock` across the whole walk (a tag-dispatched `TView` ctor snapshots under the already-held lock to avoid re-locking the non-recursive mutex; no walker machinery re-acquires `DataLock`, and `~TView`/`AcquireCurrentMapping` take only `MappingLock`, preserving the established `DataLock → MappingLock` order, so it cannot deadlock). Verified by re-running the TSan-instrumented `orlyi` under the concurrent-writer bench: the `GetLowestUpdate` race site no longer appears. (closes #237, 2026-06-29)
- **O(N²) writes, and the dormant-merge bug cascade it uncovered** (#227). Writing N keys to a POV was O(N²): the mem→disk merge was enqueued only on the empty→non-empty edge, so under sustained writes the memory layer never drained, grew unbounded, and every write paid an O(layer-size) linear scan. The root cause was an **inverted merge-scheduler deadline** (`deadline < now` where it needed `>`) that meant `StepMergeMem`/`StepMergeDisk` *never ran* — so the entire LSM merge / disk-compaction subsystem had been dormant since the 2014-era codebase, and fixing the scheduler exposed a chain of latent correctness and stability bugs that had never been exercised. All fixed here, each gated on write-then-read-back (distinct keys *and* repeated `+=`, single- and multi-POV): **(1)** the scheduler deadline itself, so merges run and the per-write cost is flat (~2 ms/write, measured flat to 8 000 writes); **(2)** the disk compaction fold discarded its computed mutator — `TUpdate::NewUpdate`'s map constructor force-tags every entry `Assign`, so a commutative key compacted across several files became `Assign(partial)` and the latest-assign-wins read fold dropped all but one partial (a counter incremented N times read back as a fraction); now folded `Add`/`Or`/… entries keep their mutator and stay summable; **(3)** the mem-layer consolidation and **(4)** the `ReleasedUpTo` publish had the same mutator-drop / a child-drop race; **(5)** only the root POV now flushes to disk (a child's zero-`UpdateId` disk copy otherwise escaped the cross-repo dedup and double-counted), and the disk present-walker replays a key's history so a merged-to-disk counter reads its full value; **(6)** every WebSocket call leaked one fiber frame (`TJumpRunnable::Main` never freed it), capping an instance at ~`max_parallel_frames` writes; **(7)** the on-disk update index was sized by *update count* rather than *distinct sequence number*, so compaction and the fold pass reserved disk blocks they never wrote and aborted (`allocated blocks that did not get written to`) at a few thousand distinct keys; **(8)** `~TDiskLayer` let a non-shutdown exception escape a `noexcept` destructor during layer cleanup, calling `std::terminate`; **(9)** the atom-layer value serialization (`TCore::NewState`/`Remap`) had no case for the recursive-variant `SelfRef` back-reference leaf (#115) — it threw `TCorrupt` when flushing any stored recursive variant, since that traversal had never run; **(10)** the arena-note comparator asserted non-empty `Tuple`/`Record`, crashing on a stored empty record (e.g. a no-payload variant case). (8)–(10) surfaced only via the example demos once the flush ran end-to-end (`recursive-variants`, `grc20-pov`). The benchmark ([`examples/agent-swarm/benchmark.py`](examples/agent-swarm/benchmark.py)) is the perf reproduction; its verdict flips from `GROWING` to `FLAT`. Validated across all eight example demos (write-then-read), the `kit2`/indy-disk unit tests, and `lang_test.py`. (closes #227, 2026-06-24 → 2026-06-25)
- **Python client decouples the per-call recv timeout from the connect timeout** (#224). `connect()` passed a single timeout to `create_connection`, which the `websocket-client` library reuses as the socket timeout for every subsequent `recv()` — so a method reply that took longer than the (short, connect-sized) window killed an otherwise-fine call, intermittently redding CI on demos unrelated to a PR's change (e.g. a `grc20-pov` read during the `examples/*` job). The Go and TS clients block indefinitely on a reply and never had this timeout, so they are unaffected. (#224, 2026-06-24)

### Changed
- **Commutative writes no longer read the current value.** A defer-safe commutative `*<[k]>::(T) OP= v` mutation compiled its LHS to `ReadOrIdentity`, which did a full flux read (`TContext::operator[]`) — but the codegen consumes only `.GetAddr()`, the value was always discarded, and an absent key already auto-initialises from the monoid identity at fold time, so the read was pure overhead. Since `TContext::operator[]` is O(pending-writes), a transaction of N commutative writes was **O(N²)**, and even a single write paid the cost. The codegen now emits a read-free `IdentityAddr` for the commutative-upsert LHS: **single writes are ~2× faster (3.6 → 1.8 ms under `--mem_sim`)** and a **batched 200-edge transaction goes 39 → 3.1 ms/edge (~12×), with scaling O(N²) → ~flat O(N)** — which unblocks bulk load. It also removes a spurious read-dependency on a coordination-free write (#49). (2026-06-26)
- **Committing a large commutative transaction is no longer O(N²).** With the per-write read gone (above), the dominant cost of a big batched write moved to the commit: `TSession::Try` registers each defer-safe commutative mutation with `TUpdate::AddEntry`, which `ReverseInsert`s it into the update's `TKey`-ordered entry collection. The deferred entries arrive in *write* order, so across several indices each insert scanned O(N) to find its slot — making a single transaction of N commutative writes **O(N²)** to commit. The entries are now sorted by `TKey` before the `AddEntry` loop, so each insert appends in O(1) (O(N log N) total); correctness is unchanged because `ReverseInsert` always finds the right slot regardless of order — only the scan length depended on it. A single-transaction bulk load now commits in **~flat O(N)**: a fresh 2 000-edge batch is ~0.33 ms/edge, holding constant from 200 to 2 000 edges (previously quadratic). This is the second and last O(N²) on the batched-write path; together with the read-elimination above, single-transaction bulk load is now linear end-to-end. (2026-06-26)
